### 1. 热榜的实现方式，引导滑动窗口热榜

答：滑动窗口热榜可以把最近一段时间切成多个小窗口，每个窗口用 Redis ZSet 记录视频热度，再用 `ZUNIONSTORE` 合并最近 N 个窗口得到热榜。这样旧热度会自然过期，新热点能快速上榜。

### 2. Redis宕机了怎么办，再启动呢

答：先降级读写权威存储或本地缓存，限制回源流量；恢复后预热热点 key，核对数据一致性，并通过监控和告警确认缓存命中率恢复。

### 3. 如何保证Redis和数据库的一致性

答：常见方案是更新数据库后删除缓存，配合重试、延迟双删或消息补偿。核心目标是控制脏数据窗口，并保证失败后可补偿。

### 4. 如果Redis宕机了，还有大量访问怎么办？

答：先降级读写权威存储或本地缓存，限制回源流量；恢复后预热热点 key，核对数据一致性，并通过监控和告警确认缓存命中率恢复。

### 6. 现有学生表（Students）和科目成绩表（Scores）。请编写一个 SQL 查询，统计总分严格大于 300 分的学生人数。要求在处理过程中，先得出每位学生的各科总成绩并按总分从大到小进行排序，最终输出符合上述条件的统计结果。

答：先按学生分组聚合总分，再用 `HAVING SUM(score) > 300` 过滤，最后对结果计数。核心 SQL 是 `SELECT COUNT(*) FROM (SELECT student_id FROM Scores GROUP BY student_id HAVING SUM(score)>300) t`。如果必须关联学生表，可以先 join 再 group by。

### 7. 三个线程分别打印a、b、c，多线程打印abc五遍

答：用三个 channel 或三个条件变量控制顺序。A 打印后通知 B，B 打印后通知 C，C 打印后进入下一轮通知 A，循环 5 次。Go 里可以用 `chan struct{}` 串联 `a -> b -> c -> a`。
